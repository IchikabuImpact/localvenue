# Claude Code 導入前のリファクタリング提案

最終更新: 2026-05-22

## 目的
エージェント向けの Markdown ドキュメントを増やす前に、誤解や運用事故を減らすための「小さく効果の高い」リファクタリング候補を整理します。

## コードベース上の現状リスク

1. **ユーティリティロジックの重複**
   - `jstTodayYmd`、タイムスタンプ付きログ、子プロセス実行ラッパー、件数制限付き並列ループが複数バッチに重複しています。
2. **バッチ契約（入出力・終了コード）の曖昧さ**
   - `daily-result-batch.js` の 2/3 スキップ終了コードなど、一部は運用されていますが、全スクリプトで明文化されていません。
3. **ページ生成処理の責務混在**
   - `generate-daily-pages.js` に SQL、会場コード補完、HTML 描画、ファイル出力が同居しています。
4. **運用ドキュメントと実装の乖離リスク**
   - スクリプトの改名・引数変更時に README / spec / 実装の整合が崩れやすい状態です。

## 優先リファクタリング計画（推奨）

### P1. バッチ共通ヘルパーの抽出（`lib/batch-utils.js`）
**狙い:** 重複排除と挙動の統一。

対象:
- 以下ヘルパーを共通モジュールへ移動
  - `jstTodayYmd`
  - `exists`
  - `runNode` / `runCaptureNode` / `runNodeWithCode`
  - `eachLimit`
  - タイムスタンプ付きロガー

効果:
- 変更時の修正箇所を減らせる
- Claude Code / 人間双方で共通前提を持ちやすくなる

### P2. スクリプト契約の明文化（`docs/SCRIPT_CONTRACTS.md`）
**狙い:** オーケストレーションを安全かつ機械可読にする。

対象:
- `scripts/` 配下の各スクリプトについて以下を定義
  - 必須引数
  - 任意引数 / 環境変数
  - 出力先（DB テーブル / ファイル / stdout）
  - 終了コードの意味（0/1/2/3 など）

効果:
- エージェントによる誤用を減らせる
- CI で契約逸脱を検知しやすくなる

### P3. `generate-daily-pages.js` の層分割
**狙い:** テスト容易性の向上と単一ファイル肥大化の解消。

分割案:
- `lib/pagegen/repositories.js`（SQL アクセス）
- `lib/pagegen/renderers.js`（HTML 描画）
- `scripts/generate-daily-pages.js`（実行オーケストレーション）

効果:
- 描画ロジックをユニットテストしやすくなる
- SQL 変更とテンプレート変更を分離レビューできる

### P4. 会場コード補完定義の一元化
**狙い:** ハードコードの重複と表示揺れの防止。

対象:
- `generate-daily-pages.js` 内の補完マップを共通化
  - 可能なら seed データで DB 側に保証
  - または `docs/VENUE_CODES.md` を元に JSON 資産を生成

効果:
- スクリプト間で会場名の不整合を防げる

## 推奨実施順
1. P1 共通ユーティリティ抽出
2. P2 スクリプト契約定義（スクリプト別テーブル）
3. P3 ページ生成の層分割
4. P4 会場コード定義の一元化

## 完了条件（事前リファクタリングフェーズ）
- `jstTodayYmd` の重複実装がなくなっている
- 少なくとも `daily-yosou-batch.js` / `daily-result-batch.js` / `generate-daily-pages.js` が共通ユーティリティを利用している
- npm スクリプト公開対象と日次バッチ対象の契約が文書化されている
- README の既存実行コマンドが変更後も有効である
